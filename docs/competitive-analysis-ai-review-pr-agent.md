# Как повысить эффективность нашего ревью: разбор ai-review и pr-agent

> Статус: **анализ, без реализации.** Дата: 2026-07-08.
> Сравниваются три инструмента: наш **ai-reviewer** (Go, GitLab-only, локальный, Claude CLI),
> **ai-review** (`xai-review`, Python, 6 VCS, single-shot) и **Qodo PR-Agent** (`pr-agent`, Python, LiteLLM, multi-tool).
> Источник данных — прямое чтение исходников всех трёх репозиториев, не README/маркетинг.
> Цель документа — вычленить, чего в нашем сервисе не хватает по части **качества анализа**, **тулинга** и **AI-агентских фич**, и где у конкурентов слабые места, которые мы уже обошли или которых надо избегать.

---

## 1. TL;DR / вердикт

По **глубине и защите от галлюцинаций** наш ai-reviewer уже сильно впереди обоих:
у нас multi-pass fan-out + skeptic-refute + детерминированные верификаторы + Go владеет позициями строк.
ai-review — чистый single-shot без верификации, позиции строк «принадлежат модели».
PR-Agent — single-shot по дизайну, self-reflection есть только в `improve`, а сам `review` вообще не якорит findings к строкам.

Но по **широте продукта** мы отстаём. Три категории того, чего у нас нет, а у них есть и это реально полезно:

1. **Выходные артефакты помимо inline-findings** — `describe` (авто-описание MR + walkthrough + Mermaid-диаграмма), `improve` (готовые committable code suggestions с dual-scoring), `update_changelog`, `generate_labels`. Мы умеем только «находить проблемы», не «помогать оформить MR».
2. **Интерактивность** — `ask`/`ask_line` (вопрос по PR или по конкретной строке), reply-в-тред (ai-review дописывает ответы в существующие треды ревью). У нас поток строго односторонний: proposed → published.
3. **Проектные правила как файлы в репозитории** — PR-Agent читает `AGENTS.md`/`CLAUDE.md` как контекст, поддерживает `best_practices`, `pr_compliance_checklist.yaml`, path-instructions. У нас есть только внутренняя «review memory» в SQLite — она не версионируется вместе с кодом и не видна команде.

Плюс несколько точечных заимствований по качеству анализа (§4) и тулингу (§5), которые стоят дёшево.

Порядок приоритетов — в §8. Коротко: сначала `describe`+`improve` (наибольшая видимая польза на существующей инфраструктуре), затем проектные rule-файлы, затем интерактивный `ask`.

## 3. Что каждый делает лучше нас (и стоит взять)

### 3.1 PR-Agent

- **`describe`** (`pr_agent/tools/pr_description.py`). Генерирует заголовок, тип PR, семантический walkthrough по файлам, **Mermaid-диаграмму изменений** (`enable_pr_diagram`), сворачиваемый список файлов. Обрабатывает большие PR несколькими async-вызовами (`max_ai_calls=4`, `extend_uncovered_files`). Для нас это самый очевидный «второй продукт» на той же инфраструктуре: у нас уже есть summary внутри ревью-объекта — его надо просто уметь постить как описание MR (через GitLab `PUT /merge_requests`).
- **`improve`** (`pr_agent/tools/pr_code_suggestions.py`). Committable code suggestions. Ключевая деталь, которую надо перенять: **номера строк генерирует не первый проход, а обязательный self-reflection-проход** (`pr_code_suggestions_reflect_prompts.toml`), который заодно скорит каждый suggestion 0–10 и жёстко занижает шумные категории (docstrings/типы/unused imports → 0). У нас в схеме уже есть `Finding.Suggestion`, но он не рендерится в GitLab suggestion-блок (` ```suggestion `). Дёшево добавить.
- **Проектные правила из репозитория**: `repo_context.py` тянет `AGENTS.md`/`CLAUDE.md`, `pr_compliance_checklist.yaml` — декларативный список правил (naming, no dead code, SRP, early-return) с success/failure-критериями. Это версионируется вместе с кодом и видно команде — в отличие от нашей SQLite-memory.
- **PR compression algorithm** (`pr_agent/algo/pr_processing.py`). Ранжирует файлы по основному языку PR, сортирует по токенам, выкидывает delete-only хунки, набирает патчи до порога бюджета. У нас per-file гранулярность спасает inline-режим, но для целостных проходов (contracts, completeness) единого ранжирующего компрессора нет.

### 3.2 ai-review (xai)

- **Reply-режимы** (`runner/inline_reply.py`, `runner/summary_reply.py`). AI **дописывает ответы в существующие треды**, а не только создаёт новые комментарии. Тред-фильтрация по magic-тегам. Для нас это мост к интерактивности без полноценного чата.
- **9 режимов рендеринга diff** (`services/diff/renderers.py`) — от «только добавленные строки» до «весь файл со снапшотом и маркерами `# added`/`# removed`». У нас рендер один (annotated diff); часть режимов (полный файл current/previous) могли бы быть полезны для отдельных пассов.
- **Hook-система** (~30 событий жизненного цикла, `services/hook/`) для метрик/нотификаций без форка. У нас есть audit-события, но нет расширяемых пользовательских хуков.
- **Built-in cost-трекинг** по токенам (`services/cost/service.py` + `resources/pricing.yaml`). У нас per-pass cost/duration персистится в `reviews.pipeline_json`, но нет агрегированного репорта стоимости.

## 4. Улучшения качества анализа (наша сильная сторона — но пробелы есть)

Даже в ядре, где мы впереди, есть чему поучиться:

1. **Ранжирующая компрессия больших MR.** Сейчас на очень большом MR целостные проходы (contracts, completeness, summary) склеивают контекст без бюджет-осведомлённого ранжирования. Портировать идею PR-Agent: сортировать файлы по релевантности (основной язык + churn + sensitive paths — у нас уже всё это посчитано в `risk.go`), набирать до бюджета, не влезшие — перечислять именами. Это устраняет тихую деградацию на монорепо-MR.

2. **Score-порог для suppressed, а не только severity.** У PR-Agent self-reflection даёт числовой score и режет по `suggestions_score_threshold`. У нас skeptic даёт categorical-вердикт (confirmed/uncertain/refuted). Числовой confidence уже есть — можно ввести настраиваемый порог показа «пограничных» findings отдельно от severity-floor.

3. **Анти-галлюцинационные формулировки промпта.** `pr_reviewer_prompts.toml:46-55` — точные инструкции «не флагай код, определённый вне diff; не спекулируй про поломку без конкретного пути выполнения». У нас `passCommonFooter` близок по духу, но стоит свериться пофразно — это дешёвая калибровка.

4. **Валидация «предложение не откатывает уже сделанное»** (`validate_one_liner_suggestion_not_repeating_code`, PR-Agent): если предлагаемый код есть в base-файле, но нет в head — score 0. Когда будем делать `improve`, эту проверку надо заложить сразу.

Важный вывод: **все три инструмента, включая коммерческий PR-Agent, делают security-ревью чисто промптом** — никто не запускает bandit/semgrep/линтеры как часть ревью. У нас единственных есть детерминированный слой верификаторов. Расширить его — это не «догнать», а **обогнать**.

## 6. AI-агентские фичи: где мы уже лидируем и куда расти

Здесь расклад нестандартный: **мы единственные с настоящим tool-use.**

- PR-Agent **явно отказался** от агентского цикла (`skills_loader.py:26-36`: «single-shot model calls and has no tool-use loop»). Их Agent Skills инлайнят весь текст SKILL.md в промпт — это не масштабируется и жрёт бюджет.
- ai-review имеет самодельный ReAct поверх голого shell (`{"action":"TOOL_CALL","command":"..."}` парсится из текста, `shell=False`, allowlist regex). Это провайдеро-агностично, но хрупко и **небезопасно**: агент исполняет LLM-сгенерированные команды с правами CI-раннера, allowlist разрешает `cat`/`git log` по любому пути → риск exfil секретов (`.env`) в промпт/артефакты.

Наш подход (claude CLI с read-only allowed_tools `Read/Grep/Glob/Bash(git diff|log|show)`, `dontAsk`, worktree в detached read-only HEAD) — и мощнее, и безопаснее.

Куда расти именно нам:

1. **MCP** — уже проработано отдельно в [gopls-mcp-analysis.md](gopls-mcp-analysis.md). Семантический (не лексический) доступ к коду для skeptic/contracts. Это наш следующий агентский рубеж, никто из конкурентов туда не идёт.
2. **Symbol-references через LSP/gopls** вместо `grep` для проверки «ломает ли изменение вызывающих» — там же.
3. **`ask_line` как агентский флоу**: у нас уже есть worktree + tools; вопрос по строке — это просто новый entrypoint поверх существующего движка, а не новая инфраструктура.

## 7. Проблемы и антипаттерны конкурентов (чего избегать)

### 7.1 ai-review (xai)

- **Позиции строк принадлежат модели без валидации.** Единственная защита — API-ошибка при постинге → fallback в общий тред. Хотя diff уже распарсен и проверка тривиальна. **Мы это уже сделали правильно** (validator + line_mapper) — не регрессировать.
- **Нет бюджета входных токенов и чанкинга.** Режимы `FULL_FILE_*` + context/summary склеивают неограниченный объём → превышение окна без graceful degradation. «Повезёт влезть».
- **`bounded_gather` глотает исключения** (`libs/asynchronous/gather.py`), сбой постинга отдельного файла → warning, общий статус «успех». Частичный провал незаметен в CI.
- **Агент исполняет shell-команды LLM** с широким allowlist → риск exfil секретов.
- **Single-shot без верификации/ранжирования** — контроль шума только тупой обрезкой первых N.
- **Хардкод версии** `xai-review==0.68.0` в двух местах (`action.yml` + `pyproject.toml`) → рассинхрон при релизе.

### 7.2 PR-Agent (Qodo)

- **`review` вообще без верификации findings**: номера строк от LLM идут только в ссылки, битая ссылка молча проглатывается (`convert_to_markdown_v2` ловит исключение и продолжает).
- **Захардкоженный `CLAUDE_MODEL = "claude-3-7-sonnet-20250219"`** для подсчёта токенов (`token_handler.py:56`) — устаревшая модель для всех Claude. Таблица `MAX_TOKENS` на ~300 моделей обновляется вручную. (Наш урок: не хардкодить версии моделей — см. `pin-model-versions` в конфиге.)
- **Парсинг YAML из LLM** — центральная точка отказа (`try_fix_yaml` с эвристиками; при неудаче тихий пустой review). Мы используем `--json-schema` у claude CLI + fallback-экстракцию — надёжнее, но `try_fix_yaml` показывает, насколько это хрупкий класс проблем.
- **GitHub inline-fallback дорогой и racy**: PENDING-review на каждый комментарий с `time.sleep(1)` → секунды и лишние API-запросы.
- **Глобальный мутабельный стейт** (Dynaconf `.set()`, мутация `os.environ` для AWS-кредов под `asyncio.Lock`) — опасно в concurrent-webhook-контексте.
- **`chat_completion` на ~230 строк** с ветвлением под каждого провайдера — высокая сложность, тяжело тестировать. (Цена универсальности LiteLLM; наш claude-CLI-only проще, но менее гибок.)
- **Заглушки open-source версии**: best-practices (`relevant_best_practices=""`), auto-best-practices, `help_docs` отключён из-за уязвимости валидации clone-target (issue #2445). Заявленное ≠ реализованное — при копировании фич смотреть код, не README.

### 7.3 Общий урок

Оба конкурента жертвуют **надёжностью позиционирования и верификацией** ради **широты интеграций**. Наш проект сделал противоположный выбор — и это правильный фундамент. Расширяться надо **не размывая** инварианты (§ CLAUDE.md): Go владеет позициями, no auto-publish, findings только на changed lines.

## 8. Дорожная карта (приоритеты, без реализации)

Отсортировано по отношению «видимая польза / стоимость», при условии сохранения инвариантов.

### Волна 1 — выходные артефакты на существующей инфраструктуре (высокая польза, низкая стоимость)

1. **`describe` — авто-описание MR.** У нас уже есть summary/risk/walkthrough-данные внутри ревью. Нужен новый service-метод + GitLab `PUT /merge_requests/:iid` (описание) и опциональная Mermaid-диаграмма. Обязательно: draft-режим и та же typed-confirmation философия, что для комментариев (не переписывать описание автора молча).
2. **`improve` — committable code suggestions.** Рендерить существующий `Finding.Suggestion` в GitLab-блок ` ```suggestion `. Обязательно портировать **self-reflection-scoring для номеров строк** и проверку «не откатывает уже сделанное» (§4.4). Владелец позиций — по-прежнему Go.

### Волна 2 — проектные правила как файлы (средняя стоимость, стратегическая ценность)

3. **Rule-файлы в репозитории**: читать `.ai-reviewer.yaml`/`AGENTS.md`/`CLAUDE.md` из worktree как контекст пассов (у нас worktree уже есть). Добавить declarative compliance-checklist по мотивам `pr_compliance_checklist.yaml`. Мигрировать часть «review memory» в версионируемый формат.

### Волна 3 — интерактивность

4. **`ask_line` / reply-в-тред.** Новый entrypoint поверх существующего движка + worktree + tools. Reply-режим ai-review (`*-reply`) как образец тред-механики. Сохранять no-auto-publish: ответы — тоже draft до подтверждения.

### Волна 4 — тулинг-дифференциаторы

5. **Расширить верификаторы**: `golangci-lint`/`ruff`/`eslint` (annotate-only), затем `semgrep` как детерминированный security-якорь. Это уникальное преимущество — никто из конкурентов статанализ не запускает.
6. **Cost-репорт в UI** (данные уже в `pipeline_json`).
7. **MCP / gopls-семантика** — по отдельному документу [gopls-mcp-analysis.md](gopls-mcp-analysis.md).

### Явно НЕ делать

- **Мульти-VCS (GitHub/Bitbucket)** — большой объём, размывает фокус «локальный GitLab-инструмент». Отложить до явного спроса.
- **Мульти-провайдер LLM / LiteLLM** — противоречит инварианту «claude CLI, все флаги config-driven». Абстракция `Client` уже есть; второй провайдер добавляется точечно, не через litellm-монолит.
- **Auto-approve / auto-merge** — запрещено инвариантами. У PR-Agent он закомментирован не случайно.
- **Самодельный ReAct поверх shell** (как в ai-review) — у нас уже есть более безопасный claude-CLI tool-use.
