# gopls MCP-сервер и ai-reviewer: анализ применимости

> Статус: **анализ, без реализации.** Дата: 2026-07-08.
> Источник новости: [go.dev/gopls/features/mcp](https://go.dev/gopls/features/mcp).
> Данные о наборе инструментов получены **эмпирически** из локально установленного `gopls v0.22.0` (Go 1.26.5), а не из веб-документации — см. раздел «Инвентаризация инструментов».

---

## 1. TL;DR / вердикт

Встроенный MCP-сервер gopls даёт LLM-агенту **семантическое** понимание Go-кода (через language server) вместо **лексического** (grep/read), которым ai-reviewer пользуется сегодня в agent-mode. Для инструмента ревью это попадает точно в две самые тяжёлые задачи:

1. **«Ломает ли изменение вызывающих?»** — `go_symbol_references` даёт точные, типизированные ссылки на символ. `grep` этого не умеет (не разрешает алиасы импортов, dot-import, множества методов, путает одноимённые символы, ловит совпадения в строках/комментариях).
2. **«Это реально баг / оно вообще компилируется?»** — `go_diagnostics` даёт ошибки парсинга/сборки и анализаторов уровня language server, богаче и быстрее (тёплый инкрементальный кэш), чем разовый `go build`, который уже есть в верификаторе.

Плюс новая возможность-репорт: `go_vulncheck` (govulncheck по достижимым из диффа зависимостям) идеально ложится на существующий паттерн «quality reports».

**Но** это улучшение стоит денег и осложнений: gopls на каждом worktree должен type-check'нуть весь модуль (холодный старт — секунды/минуты и память), а пайплайн делает **веерный fan-out** множества вызовов `claude`. Наивное включение = N холодных стартов gopls на один head SHA. Это управляемо (см. §7), но требует продуманной модели процессов и строгого списка разрешённых tools, чтобы не нарушить инварианты проекта.

Рекомендация — поэтапно, начиная с самого дешёвого и безопасного (детерминированный vulncheck-репорт + gopls-диагностика в слое верификаторов), затем — agent-facing MCP только для проверочных проходов (skeptic/contracts) с одним «тёплым» gopls на worktree. Подробности в §8.

## 2. Что это на самом деле (проверено эмпирически)

gopls начиная с **v0.20** содержит экспериментальный встроенный MCP-сервер. Два режима запуска:

| Режим        | Команда                                  | Транспорт | Видит                                      | Для ai-reviewer                                                                                                                                                     |
| ------------ | ---------------------------------------- | --------- | ------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **detached** | `gopls mcp`                              | stdio     | только файлы **на диске**                  | ✅ единственно релевантный: ревью идёт по зафиксированному head SHA в worktree                                                                                      |
| **attached** | `gopls serve -mcp.listen=localhost:PORT` | HTTP+SSE  | + несохранённые буферы активной LSP-сессии | «несохранённые буферы» **нерелевантны**, но HTTP-режим даёт _общую тёплую сессию_ на несколько параллельных проходов — это как раз то, что нужно fan-out'у (см. §7) |

Ключевой вывод для нашего кейса: фича «видит несохранённые изменения в буфере» (главный аргумент attached-режима в новости) для ai-reviewer **бесполезна** — мы всегда работаем с уже закоммиченным кодом. Зато у attached/serve-режима есть _побочное_ свойство — одна прогретая type-check-сессия, разделяемая между многими клиентами, — и вот оно для нас ценно.

Требования: `gopls v0.20+` (`go install golang.org/x/tools/gopls@latest`). Локально установлен **v0.22.0**.

### Модель доверия gopls (важно для инвариантов проекта)

gopls в роли MCP-сервера делает ровно то, что делает в обычной IDE-сессии: читает файлы, запускает `go`-команды (`go list`, загрузка пакетов, type-check), ходит в module proxy и в БД уязвимостей, пишет в свой кэш. Он **не** пишет напрямую в дерево исходников (правки только возвращаются клиенту как diff) и **не** делает произвольных сетевых запросов. Это примерно уровень доверия `go_build`/`go_vet` из нашего проекта, **кроме двух инструментов**: `go_rename_symbol` (генерирует правки — write-ориентированный) и `go_vulncheck` (сеть к proxy/vuln-DB).

## 3. Инвентаризация инструментов (gopls v0.22.0, из `tools/list`)

Получено напрямую из бинарника (`gopls mcp` → MCP `tools/list`). В v0.22 ровно **8 инструментов** (в разных версиях набор отличается — веб-доки перечисляют до 12; набор помечен как экспериментальный и меняется между релизами — это важное следствие для конфигурируемости, см. §7-E):

| Инструмент             | Что делает                                                                                                       | Тип             | Ценность для ревью                                                                   |
| ---------------------- | ---------------------------------------------------------------------------------------------------------------- | --------------- | ------------------------------------------------------------------------------------ |
| `go_workspace`         | Резюме структуры воркспейса (module / workspace / GOPATH)                                                        | read            | низкая (разовая ориентация)                                                          |
| `go_search`            | Fuzzy-поиск символов по воркспейсу и зависимостям (регистронезависимо, ≤100 результатов)                         | read            | **средняя** — замена/дополнение FTS для Go-символов                                  |
| `go_file_context`      | Резюме **межфайловых зависимостей внутри пакета** (какие декларации других файлов пакета использует данный файл) | read            | **средняя** — семантическая замена FTS-эвристики «related files»                     |
| `go_package_api`       | Резюме публичного API пакета (экспортируемые функции/типы + doc-комментарии)                                     | read            | **средняя–высокая** — компактный контекст для проходов «contracts» и по зависимостям |
| `go_symbol_references` | Точные локации ссылок на символ уровня пакета (в т.ч. из импортируемого пакета: `lib.Bar`)                       | read            | **высокая** — impact-анализ вызывающих, точнее grep                                  |
| `go_diagnostics`       | Ошибки парсинга/сборки по всему воркспейсу + линтинг для указанных активных файлов                               | read/analyze    | **высокая** — LSP-grade замена/усиление `go_build`/`go_vet`                          |
| `go_rename_symbol`     | Переименование символа по воркспейсу (возвращает правки)                                                         | **write**       | ❌ не нужен и противоречит «никаких правок»                                          |
| `go_vulncheck`         | govulncheck по паттерну пакетов (по умолчанию `./...`)                                                           | read + **сеть** | **новый репорт** — уязвимости в достижимых зависимостях                              |

Официальные model-instructions (`gopls mcp -instructions`) описывают два рабочих цикла: **Read** (`go_workspace` → `go_search` → `go_file_context` → `go_package_api`) и **Edit** (перед правкой символа — `go_symbol_references`; после правки — обязательный `go_diagnostics`; при смене зависимостей — `go_vulncheck`). Для ai-reviewer релевантен по сути только Read-цикл плюс диагностика как проверка.

## 4. Как ai-reviewer понимает код сегодня (релевантные поверхности)

Кратко, по результатам разбора кода (ссылки — на реальные места):

- **Agent-mode = ФС-доступ к read-only worktree на head SHA.** `claude` запускается с `cmd.Dir = worktree` и фиксированным списком read-only tools: `Read, Grep, Glob, Bash(git diff/log/show *)` — см. [claude_cli.go:103-117](internal/llm/claude_cli.go#L103-L117), дефолты в [internal/config/config.go](internal/config/config.go). MCP **нигде не используется** (репозиторный grep по `mcp` — 0 совпадений).
- **Investigation-протокол**: [prompts.go](internal/review/prompts.go) (`writeInvestigationSection`) прямо инструктирует агента _«grep'ай вызывающих изменённых экспортируемых символов, проверяй интерфейсы/контракты, читай полный файл»_. Это ровно те задачи, где семантика gopls бьёт grep.
- **Related files** подбираются FTS-эвристикой: [internal/service/context.go](internal/service/context.go) (`findRelatedFiles`) строит BM25 OR-запрос по частым идентификаторам из изменённых строк. Лексика, не семантика.
- **Skeptic** ([internal/review/skeptic.go](internal/review/skeptic.go)) — agent-mode вызов, который пытается **опровергнуть** находки, читая реальный код worktree (`Read/Grep/Glob/git`). Это главный кандидат на усиление семантикой.
- **Verifier-плагины** ([verifier_go_build.go](internal/review/verifier_go_build.go), [verifier_go_vet.go](internal/review/verifier_go_vet.go), [verifier_gotest.go](internal/review/verifier_gotest.go)) детерминированно запускают Go-тулчейн и могут **снимать** ложные claim'ы о невозможности компиляции (`go_build` → `VerdictDrop` на чистой сборке). Это детерминированный слой, не LLM.
- **Quality reports** (риск, completeness, coverage) — репорты, а не находки на строках; персистятся как `reviews.*_json`. Естественное место для нового vulncheck-репорта.
- **Инварианты** (CLAUDE.md): Go владеет валидацией и позициями (не модель); верификаторы по умолчанию **не исполняют код репозитория**; всё, что исполняет код или ходит в сеть, — **явный opt-in**; никаких правок MR; localhost-only.

## 5. Ключевая идея: две разные точки интеграции

Важно не смешивать два принципиально разных способа применить gopls. Они дают разную ценность и по-разному рискуют:

### 5A. Agent-facing (gopls как MCP-сервер для LLM)

gopls-инструменты добавляются в allow-list `claude` (`mcp__gopls__go_symbol_references`, …). Улучшают **качество расследования модели**: агент вместо grep получает точные ссылки, компактный API-summary, LSP-диагностику. Это вход в _рассуждение модели_ — ровно как grep/Read сегодня, поэтому **не** угрожает инварианту «Go владеет позициями/валидацией» (детерминированный validator/line-mapper по-прежнему решает всё про позиции). Риски — стоимость (§7) и модель доверия (нужен строгий whitelist tools).

### 5B. Deterministic-facing (gopls/govulncheck из кода проекта, без LLM)

Проект сам вызывает gopls (в CLI/API-режиме) или `govulncheck` напрямую и кормит результат в **детерминированный слой** — новый верификатор `gopls_diagnostics` и/или новый репорт `vulncheck_json`. Не зависит от модели, максимально доверенный сигнал, ложится на существующий паттерн верификаторов/репортов. Для этой поверхности **MCP формально не нужен** — ценность новости здесь в том, что «gopls быстрый и резидентный», а не в MCP-протоколе. Но именно `go_diagnostics` показывает, что gopls уже умеет отдавать эти сигналы в удобной форме.

Практический вывод: **5B дешевле и безопаснее, начинать стоит с него**; 5A даёт больший потолок качества, но требует решить проблему стоимости fan-out.

## 6. Возможности: инструмент → конкретное улучшение

Отранжировано по ожидаемой отдаче.

### 6.1. `go_symbol_references` → точный impact-анализ вызывающих ⭐ (высокая)

- **Проблема сейчас**: investigation-протокол просит агента `grep` вызывающих. Grep лексический: пропускает алиас-импорты (`import f "…/foo"`), dot-import, ловит одноимённые символы из других пакетов, совпадения в строках/комментариях. Для вопроса «изменённая сигнатура/поведение экспортируемой функции сломает вызывающих?» это источник и ложных срабатываний, и пропусков.
- **С gopls**: `go_symbol_references({file, name})` возвращает типизированные ссылки по всему воркспейсу (включая `lib.Bar` из импортируемого пакета). Прямо усиливает correctness/contracts-проходы и skeptic.
- **Где**: [prompts.go](internal/review/prompts.go) `writeInvestigationSection` (сменить формулировку с «grep callers» на «используй `go_symbol_references` для Go-символов»), доступно в проходах и в [skeptic.go](internal/review/skeptic.go).

### 6.2. `go_diagnostics` → усиление детерминированного верификатора ⭐ (высокая)

- **Проблема сейчас**: `go_build` verifier ловит только claim'ы о невозможности компиляции и запускает полный `go build`/`go test -c` на каждый такой claim (кэш по пакету, таймаут 120с). `go_vet` — отдельный запуск.
- **С gopls**: `go_diagnostics` отдаёт ошибки парсинга/сборки + анализаторов из резидентного language server — быстрее (инкрементально, один прогрев воркспейса на все запросы) и богаче (набор анализаторов gopls шире, чем `go vet`). Два применения:
  - **5B (детерминированно)**: новый верификатор `gopls_diagnostics` — на чистой диагностике снимать ложные compile-claim'ы (как `go_build`), а реальные диагностики файла — аннотировать (как `go_vet`). Уровень доверия — как у `go_build`/`go_vet` (type-check, без исполнения тестов) → **можно в дефолтный безопасный tier**, с той же оговоркой «любой сбой окружения ⇒ Keep», что уже есть в [verifier.go](internal/review/verifier.go).
  - **5A (агентом)**: skeptic может подтверждать/опровергать баг-claim'ы, спрашивая `go_diagnostics` по файлу, вместо запуска `go build` вручную.
- **Тонкость доверия**: gopls type-check запускает `go list`/загрузку пакетов; это не исполнение бизнес-кода репозитория (в отличие от `go_test`), но с cgo/`//go:generate` в общем случае граница тоньше — стоит подтвердить эмпирически (§10). По умолчанию — тот же tier, что `go_build`.

### 6.3. `go_vulncheck` → новый quality-report по безопасности ⭐ (высокая, но opt-in)

- **Идея**: у проекта есть security-специализированный проход и репорты (risk/completeness/coverage). `go_vulncheck` даёт govulncheck — **достижимые** из изменённого кода известные CVE в зависимостях. Особенно ценно, когда дифф добавляет/бампает зависимость (эвристика по изменению `go.mod`).
- **Куда ложится**: новый детерминированный репорт `reviews.vulncheck_json` рядом с risk/completeness/coverage — репорт, не находка на строке (уязвимость не якорится к изменённой строке). Паттерн уже описан в CLAUDE.md.
- **Ограничение**: ходит в сеть (vuln-DB/proxy). По правилам проекта → **явный opt-in**, как `coverage.enabled` и `go_test`. Гейтить на «дифф трогает `.go`/`go.mod`».

### 6.4. `go_package_api` → компактный контекст контрактов/зависимостей (средняя–высокая)

- **Проблема сейчас**: чтобы понять API вызываемого пакета (свой монорепный или сторонний), агент читает целые файлы; бюджет full-file ограничен ([context.go](internal/review/context.go) `ContextBudget`, 256 КБ).
- **С gopls**: `go_package_api({packagePaths})` — компактное резюме экспортируемого API + doc-комментарии. Токен-эффективнее полного файла и точнее для вопроса «правильно ли дифф использует эту функцию/сигнатуру?». Снимает давление на full-file-бюджет и помогает проходу «contracts».

### 6.5. `go_file_context` / `go_search` → семантическая замена FTS «related files» (средняя)

- **Проблема сейчас**: [findRelatedFiles](internal/service/context.go) — BM25 по совпадению идентификаторов (лексика). Даёт «шумные» лиды.
- **С gopls**: `go_file_context` — _реальные_ межфайловые зависимости внутри пакета; `go_search` — fuzzy-поиск символа по воркспейсу+зависимостям. Семантически точнее.
- **Важная оговорка**: FTS остаётся нужен — ревьюер **полиглотный** (в проекте есть `py_syntax`, `tsc` верификаторы). gopls помогает только Go-файлам. То есть это дополнение для Go-диффов, не замена FTS вообще.

### 6.6. Что НЕ применять

- `go_rename_symbol` — write-ориентированный (правки-рефакторинг). Противоречит инварианту «никаких правок MR» и ревью-задаче в принципе. **Исключить из allow-list.**
- attached-режим _ради_ несохранённых буферов — для ревью закоммиченного кода бессмысленно (serve-режим полезен по другой причине — §7).

## 7. Ограничения, риски и стыковка с инвариантами

### 7-A. Стоимость и жизненный цикл процесса gopls (главный риск)

Каждое ревью — свежий worktree на head SHA. gopls (detached) на первый tool-call делает **холодный type-check всего модуля** — секунды на среднем репо, до минут и много памяти на большом монорепо. Пайплайн же делает **fan-out**: 5 проходов + батчи skeptic + completeness → десятки процессов `claude`.

- **Наивно (detached, per-claude-call)**: `claude` спавнит свой `gopls mcp` (stdio, cwd=worktree) на каждый вызов → **N холодных type-check'ов одного head SHA**. Дорого.
- **Лучше (serve, один на worktree)**: ai-reviewer сам поднимает **один** `gopls serve -mcp.listen=localhost:PORT` на worktree (жизненный цикл — рядом с cleanup в [internal/git/repo_cache.go](internal/git/repo_cache.go)), прогревает и отдаёт всем проходам **один HTTP MCP-endpoint**. Все проходы делят тёплый кэш type-check. Это и есть неочевидная причина, по которой attached/serve-режим полезен нам — **не** ради буферов, а ради **амортизации прогрева между параллельными проходами**. Localhost-сокет вписывается в localhost-only-постуру проекта.
- **Гейтинг**: включать gopls только если дифф трогает `.go` (для не-Go MR не платить прогрев вообще).

### 7-B. Модель доверия / инвариант «верификаторы не исполняют код по умолчанию»

Строго ограничить allow-list безопасным read-подмножеством: `go_workspace, go_search, go_file_context, go_package_api, go_symbol_references, go_diagnostics`. **Исключить** `go_rename_symbol` (write). **`go_vulncheck` — только по явному opt-in** (сеть). MCP-инструменты в `--allowedTools` именуются `mcp__gopls__<tool>` — существующий механизм allow-list это уже покрывает.

### 7-C. Инвариант «Go владеет позициями/валидацией» — не нарушается

gopls-выход в поверхности 5A — вход в _рассуждение модели_, как grep/Read. Детерминированные validator/line-mapper по-прежнему единолично решают file-in-diff, line→position, severity, dedupe, cap. gopls ничего из этого не касается. ✅

### 7-D. Полиглотность

gopls помогает только Go. Не-Go MR (Python/TS/прочее) ничего не получают → обязательный гейт по расширениям, иначе платим прогрев впустую.

### 7-E. Экспериментальность и версии

Набор tools экспериментальный и меняется между релизами (эмпирически: v0.22 = 8 tools, часть веб-доков перечисляет 12). Следствия:

- всё должно быть **config-driven** (в духе «каждый флаг claude конфигурируем»): путь к `gopls`, вкл/выкл, список разрешённых MCP-tools, режим (detached/serve), таймаут прогрева;
- **defensive**: отсутствие/старый gopls или неизвестный tool → тихий фолбэк на текущий grep/Read-агент-мод (как worktree сейчас «best-effort»);
- проверка версии gopls (`v0.20+`) — в `doctor`, рядом с проверками GitLab/claude/git/FTS5.

### 7-F. Детерминизм ревью

Разделяемая тёплая gopls-сессия и сетевой vulncheck вносят недетерминизм/сетевую зависимость. Для детерминированных репортов (5B) это учитывать: vulncheck зависит от состояния vuln-DB на момент запуска — это нормально для security-репорта, но должно быть явно (timestamp в `vulncheck_json`).

## 9. Чего НЕ делать

- Не включать gopls глобально `claude mcp add gopls` в пользовательском `~/.claude.json` — это неконтролируемо и невоспроизводимо. Только per-invocation `--mcp-config` (+ `--strict-mcp-config`), чтобы ревью оставалось детерминированным и изолированным.
- Не давать агенту `go_rename_symbol` (write) — против инварианта «никаких правок».
- Не включать по умолчанию `go_vulncheck` (сеть) и не включать gopls для не-Go диффов.
- Не позволять gopls-выходу влиять на детерминированные позиции/валидацию — только на рассуждение модели или на детерминированные верификаторы/репорты.
- Не спавнить gopls per-claude-call на большом монорепо без serve-амортизации — это N холодных type-check'ов.

## 10. Что стоит замерить/прототипировать перед решением

1. **Холодный старт gopls** на реальных ревьюируемых репозиториях (маленький / средний / монорепо): время до первого ответа `go_diagnostics`, RSS. Это определяет, жизнеспособен ли per-call detached или обязателен serve.
2. **Амортизация**: сколько экономит один `gopls serve` на worktree против N detached-спавнов при типичном fan-out (5 проходов + skeptic-батчи).
3. **Граница доверия `go_diagnostics`**: подтвердить, что type-check gopls **не исполняет** бизнес-код репозитория (cgo/generate-случаи), чтобы обосновать дефолтный безопасный tier для `gopls_diagnostics` верификатора.
4. **Качество impact-анализа**: на наборе реальных находок сравнить точность/полноту вызывающих у `go_symbol_references` против текущего grep-подхода — это главный измеримый выигрыш качества ревью.
5. **Стабильность набора tools** между версиями gopls (защитный код + версия в `doctor`).

## 11. Дополнение: symgraph и инструменты его класса

> Оценка получена **чтением исходников** `/Users/vkuramshin/p/dev/symgraph` (~14 тыс. строк Rust, 186 тестов), а не README. Где README расходится с кодом — верю коду и отмечаю это.

### 11.1. Что это (по факту, не по README)

symgraph — компилируемый Rust MCP-сервер, который строит **граф символов** через tree-sitter для 13 языков (Rust, Go, TS/JS, Python, Java, C/C++, C#, Kotlin, Scala, Groovy, Ruby) и хранит его в SQLite (`nodes`, `edges`, `unresolved_refs`, FTS5). Отдаёт ~21 MCP-инструмент: `search`, `callers`/`callees`/`references`, `impact`, `diff-impact` (импакт по диапазону строк **или** по `git diff <ref>`), `hierarchy`, `implementations`, `path`, `unused`, плюс git-aware (`churn`, `blame`) и coupling-аналитику (`module-graph`, `coupling-score`, `god-struct`, `dispatch-sites`). Есть stdio и HTTP-транспорт, bearer-auth, инкрементальная переиндексация по content-hash, git-common-dir хранилище. Инфраструктура и безопасность (защита от path-traversal, атомарный swap индекса, constant-time сравнение токена) — сделаны добротно.

Принципиально это **тот же класс инструмента, что gopls MCP, но с обратным компромиссом**: gopls = высокая точность, только Go, нужен Go-тулчейн и type-check на каждый worktree; symgraph = широкое покрытие языков, один бинарь без тулчейнов, **но резолюция — эвристическая**.

### 11.2. Единственное стратегическое преимущество для нас: полиглотность

Ревьюер обрабатывает MR **на любых языках** (в проекте есть верификаторы `py_syntax`, `tsc`). gopls закрывает только Go. symgraph (или инструмент его класса) закрывает семантику для TS/Python/Java/C++ и т.д. **одним бинарём без установки N тулчейнов** — это ровно та дыра, которую gopls оставляет. Плюс форма `diff-impact(file, start_line, end_line)` / `diff-impact(git_ref)` буквально совпадает с формой MR-диффа (набор диапазонов строк).

Дополнительно: проект **уже спроектирован** под такой слой — в схеме есть пустые таблицы `repo_symbols`, `repo_imports`, `repo_references` и конфиг-флаги `IndexConfig{TreeSitter, LSP, VectorSearch}` (пока populate'ится только FTS). symgraph — готовая внешняя реализация того, подо что эти таблицы и заводились.

### 11.3. Честная оценка точности — где symgraph НЕ подходит (критично)

Ядро ценности ревьюера — вопрос **«сломает ли это изменение вызывающих / кто реализует этот интерфейс»**. Именно здесь symgraph слаб, и слаб принципиально:

- **Резолюция рёбер — чистое сопоставление по имени, `LIMIT 1`.** Единственная эвристика (`find_target_preferring_file`, `src/db/mod.rs:1332-1349`): «есть определение с таким именем в том же файле? бери его; иначе — первый глобальный по rowid». Имя вызова заранее обрезается до голого идентификатора (`db.Get()`, `cache.Get()`, `pkg.Get()` → все `"Get"`). Нет разрешения типов, области видимости, импортов, алиасов. Две функции `Get` в разных пакетах **схлопываются в одну**: все вызовы валятся на первую по индексации, у остальных — ноль вызывающих. Для распространённых имён (`New`, `Close`, `Read`, `Handle`, `String`) это доминирующий режим. `qualified_name` в схеме есть, но в резолюции **не используется**.
- **Рёбра `implements`/`extends` не создаются вообще.** Enum, запросы (`find_implementations`), тесты (на вручную вставленных рёбрах) есть — а **ни один путь извлечения их не эмитит** (подтверждено grep'ом по репозиторию). То есть `symgraph-implementations` и иерархия наследования на реальном коде возвращают пусто, хотя README их рекламирует. Это прямой разрыв «фича заявлена — фичи нет».
- **`diff-impact` мелкий**: находит символы, пересекающие диапазон, и добавляет **ровно один уровень прямых вызывающих** (не транзитивно, только `Calls`, cap 100). Транзитивный blast-radius и не-call зависимости (поля/импорты) он не видит.
- **Покрытие языков неравномерно.** Все языки получают `Contains`+`Calls`+`Imports`, но field-access (`accesses`/`mutates`) — только Rust/TS/JS/Python; enum-dispatch и `&mut`-coupling — только Rust. Для **Go** извлекаются лишь определения + `Contains` + `Calls` + `Imports`, без field-level рёбер. Метаданные (visibility, async/static) — сканированием подстрок первой строки, хрупко.
- **Вывод — markdown, не структура**, для самых нужных ревью-инструментов: `diff-impact`, `blame`, `churn` вообще без `format=json`; структурный JSON есть лишь у части (`search`/`references`/`module-graph`/`coupling-score`).
- **Бенчмарков нет** — поведение на большом монорепо не измерено.

**Ключевой аргумент против «в лоб».** Для ревьюера уверенно-неверный семантический ответ **хуже**, чем grep. Сейчас investigation-протокол честно говорит модели «grep лексический — снижай уверенность». Если вместо этого дать «точный» ответ графа, который на самом деле схлопнул два `Get`, модель **повысит** уверенность на ложных данных — и `skeptic`, и находки станут хуже, а не лучше. Низкоточный семантический слой на precision-critical пути способен **деградировать** качество ревью.

### 11.4. Где symgraph (или его класс) реально помог бы

Там, где **агрегация гасит неточность** и не нужна пер-ребёрная корректность — это репорты, не находки на строке:

- **Полиглотные hotspot/coupling-репорты**: `churn` (волатильность), `coupling-score`, `god-struct`, `module-graph` (fan-in/out, циклы), `dispatch-sites`. Ложатся на существующий паттерн **quality reports** (рядом с детерминированным `ComputeRisk`) — и, в отличие от gopls-диагностики, работают для **не-Go** MR. Ранжирование по имени терпимо к схлопыванию имён (оно усредняется).
- **Быстрый символьный поиск/навигация как замена/дополнение FTS** ([findRelatedFiles](internal/service/context.go)) для языков, где нет LSP: `search` (FTS5 + семантический bm25) и `file` (символы файла) — лиды для агента, не истина.
- Как **эксперимент** для не-Go языков — там, где сейчас у агента вообще только grep, даже неточный граф расширяет охват (но подавать как «возможно связано», с той же оговоркой о низкой уверенности).

### 11.5. gopls vs symgraph vs «идеал»

| Критерий | gopls MCP | symgraph | «идеал»: пер-язык LSP через MCP |
| --- | --- | --- | --- |
| Точность резолюции | высокая (по типам) | низкая (по имени, `LIMIT 1`) | высокая |
| Покрытие языков | только Go | 8–13 (неравномерно) | сколько LSP поднимешь |
| Наследование/implements | ✅ (references/типы) | ❌ (не эмитится) | ✅ |
| Impact по диапазону строк | косвенно (references) | ✅ прямой `diff-impact` (но depth-1) | зависит |
| Стоимость на worktree | type-check модуля (дорого) | индексация tree-sitter (дёшево, инкрементально) | N тулчейнов (дорого) |
| Развёртывание | нужен Go-тулчейн | один бинарь | N серверов + тулчейны |
| Годен для precision-пути ревью | ✅ (для Go) | ⚠️ нет | ✅ |
| Годен для полиглотных репортов | ❌ (не-Go) | ✅ | ✅ |

Настоящий «идеал» под precision-критичный полиглот — не symgraph, а **набор пер-язычных LSP через MCP** (gopls + tsserver/pyright + jdtls…): точность LSP на каждом языке. Цена — N серверов/тулчейнов и оркестрация. symgraph меняет эту точность на «один бинарь, все языки» — разумный размен для _ранжирования_, но не для _корректности_.

### 11.6. Вердикт (твёрдо)

- **symgraph как есть — не брать на precision-критичный путь** (impact / «ломает вызывающих» / implementations). Name-based `LIMIT 1` и отсутствующие рёбра наследования делают его ненадёжным именно там, где ревьюеру важнее всего, а уверенно-неверный ответ вреднее grep.
- **Узкое «да» — как опциональный источник полиглотных hotspot/coupling-репортов** (churn/coupling/god-struct/module-graph) в детерминированном слое quality-reports, для не-Go MR. Цена: сопровождать внешний Rust-бинарь + индекс на worktree; выигрыш поверх уже существующего `ComputeRisk` — умеренный, но это единственное, что даёт покрытие не-Go.
- **Инструмент этого класса (семантический MCP-граф, language-agnostic) — правильное направление** для полиглотной дыры, которую оставляет gopls. Но планка точности для ревью — **уровень LSP**, а не эвристика; иначе инструмент вредит. Практический вывод: **на Go — gopls (§1–10); на прочих языках — не спешить с symgraph на путь находок, а либо ждать/поднимать пер-язычные LSP-MCP, либо использовать symgraph только для агрегированных репостов.**
